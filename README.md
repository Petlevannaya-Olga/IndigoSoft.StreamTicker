# Тестовое задание "Система агрегации и обработки данных с биржевых торгов"

**Описание:**
Разработать имитацию системы сбора, обработки и хранения ценовых данных с нескольких бирж в реальном времени. Нагрузка: до 50-100 тиков/сек суммарно.

**Функционал:**
### 1. **Сбор данных**:
-Клиент WebSocket (real-time stream)  2-3 клиента одновременно подключаются к серверным вебсокетам и получают поток котировок (тикер, цена, объем, timestamp) - формат может быть разным

### 2. **Обработка потока данных**:
- Нормализация к единому формату (тикер, цена, объем, timestamp)
- Удаление дубликатов

### 3. **Хранение в БД**:
- Raw данные (тики)
### 4. **Базовый мониторинг**:
- Логирование основных событий (подключение/отключение источника, ошибки)
- Простой счетчик обработанных тиков (в консоль или лог)

**Требования:**
- Обработка 50-100 тиков/сек (2-3 источника)
- Параллельная работа с несколькими WebSocket подключениями
- Чистая архитектура, SOLID принципы
- Обработка обрывов соединения и автоматическое переподключение

**Что оценивается:**
- Многопоточность и асинхронная обработка
- Работа с WebSocket соединениями
- Производительность и эффективность работы с БД
- Расширяемость архитектуры для добавления новых бирж
- Обработка ошибок и отказоустойчивость
- Покрытие тестами базовых сценариев

# Как запустить проект
1. Запустите IndigoSoft.StreamTicker.Wss — веб-сервер, генерирующий фейковые биржевые данные в формате Binance.
2. Запустите IndigoSoft.StreamTicker.Presentation — консольное приложение, содержащее основную бизнес-логику проекта.
3. В проекте используется база данных SQLite, поэтому дополнительная настройка не требуется. Выбор SQLite обусловлен её простотой, эффективностью и отсутствием необходимости в развёртывании отдельного сервера.

# Подключенные биржи
В проекте реализована загрузка рыночных данных в реальном времени с криптовалютных бирж Binance, Kraken и Bybit.

Помимо этого, для обеспечения стабильной и непрерывной нагрузки в системе предусмотрен механизм фейковой генерации данных, имитирующий поток котировок в формате биржи Binance. Это позволяет тестировать и отлаживать работу приложения независимо от доступности внешних источников данных, а также моделировать различные сценарии нагрузки.

# Обоснование выбора Dataflow как механизма потоковой обработки данных
Выбор модели Dataflow в качестве механизма потоковой обработки данных обусловлен её естественной ориентацией на построение конвейера обработки, где данные последовательно проходят через независимые этапы, что упрощает масштабирование, сопровождение и распараллеливание вычислений. 

В отличие от использования каналов (channels), которые представляют собой низкоуровневый механизм передачи данных между потоками и требуют явного управления синхронизацией и потоками выполнения, Dataflow обеспечивает более высокий уровень абстракции и автоматическое управление потоком данных, что делает архитектуру системы более прозрачной и удобной для развития.
