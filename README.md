# Тестовое задание "Система агрегации и обработки данных с биржевых торгов"

**Описание:**
Разработать имитацию системы сбора, обработки и хранения ценовых данных с нескольких бирж в реальном времени. Нагрузка: до 50-100 тиков/сек суммарно.

**Функционал:**
### 1. **Сбор данных**:
-Клиент WebSocket (real-time stream)  2-3 клиента одновременно подключаются к серверным вебсокетам и получают поток котировок (тикер, цена, объем, timestamp) - формат может быть разным

### 2. **Обработка потока данных**:
- Нормализация к единому формату (тикер, цена, объем, timestamp)
- Удаление дубликатов

### 3. **Хранение в БД**:
- Raw данные (тики)
### 4. **Базовый мониторинг**:
- Логирование основных событий (подключение/отключение источника, ошибки)
- Простой счетчик обработанных тиков (в консоль или лог)

**Требования:**
- Обработка 50-100 тиков/сек (2-3 источника)
- Параллельная работа с несколькими WebSocket подключениями
- Чистая архитектура, SOLID принципы
- Обработка обрывов соединения и автоматическое переподключение

**Что оценивается:**
- Многопоточность и асинхронная обработка
- Работа с WebSocket соединениями
- Производительность и эффективность работы с БД
- Расширяемость архитектуры для добавления новых бирж
- Обработка ошибок и отказоустойчивость
- Покрытие тестами базовых сценариев

# Как запустить проект
1. Запустите IndigoSoft.StreamTicker.Wss — веб-сервер, генерирующий фейковые биржевые данные в формате Binance.
2. Запустите IndigoSoft.StreamTicker.Presentation — консольное приложение, содержащее основную бизнес-логику проекта.
3. В проекте используется база данных SQLite, поэтому дополнительная настройка не требуется. Выбор SQLite обусловлен её простотой, эффективностью и отсутствием необходимости в развёртывании отдельного сервера.

# Подключенные биржи
В проекте реализована загрузка рыночных данных в реальном времени с криптовалютных бирж Binance, Kraken и Bybit.

Помимо этого, для обеспечения стабильной и непрерывной нагрузки в системе предусмотрен механизм фейковой генерации данных, имитирующий поток котировок в формате биржи Binance. Это позволяет тестировать и отлаживать работу приложения независимо от доступности внешних источников данных, а также моделировать различные сценарии нагрузки.

# Обоснование выбора Dataflow как механизма потоковой обработки данных
Выбор модели Dataflow в качестве механизма потоковой обработки данных обусловлен её естественной ориентацией на построение конвейера обработки, где данные последовательно проходят через независимые этапы, что упрощает масштабирование, сопровождение и распараллеливание вычислений. 

В отличие от использования каналов (channels), которые представляют собой низкоуровневый механизм передачи данных между потоками и требуют явного управления синхронизацией и потоками выполнения, Dataflow обеспечивает более высокий уровень абстракции и автоматическое управление потоком данных, что делает архитектуру системы более прозрачной и удобной для развития.

# Обеспечение непрерывного соединения с биржами с помощью политик (библиотека Polly)
В проекте для обеспечения непрерывного соединения с биржами используется комплекс политик устойчивости, реализованных с помощью библиотеки **Polly**. Основная идея заключается в комбинировании нескольких механизмов — **retry**, **circuit breaker**, **timeout** и **bulkhead** — которые совместно обеспечивают стабильную работу WebSocket-подключений даже при нестабильной сети или сбоях на стороне бирж.

Политика **Retry** реализует автоматическое переподключение при возникновении ошибок. Используется бесконечный цикл повторных попыток (WaitAndRetryForeverAsync) с экспоненциальным увеличением задержки и добавлением jitter (случайной составляющей). Это позволяет избежать эффекта «синхронного шторма» подключений и равномерно распределяет нагрузку при массовых реконнектах.

Политика **Circuit** Breaker предотвращает постоянные попытки подключения к заведомо недоступному ресурсу. После 5 последовательных ошибок соединение «размыкается» на 30 секунд, в течение которых новые попытки не выполняются. Это снижает нагрузку на систему и защищает от деградации при длительных сбоях на стороне биржи. После паузы происходит переход в состояние half-open, где выполняется пробная попытка восстановления соединения.

Политика **Timeout** ограничивает максимальное время выполнения одной операции (например, установления соединения или получения данных). Если операция зависает дольше заданного времени (10 секунд), она принудительно прерывается, что предотвращает блокировку потоков и зависание системы.

Политика **Bulkhead** ограничивает количество одновременно выполняемых операций (максимум 5 активных и 10 ожидающих). Это защищает систему от перегрузки в случае массовых попыток переподключения, обеспечивая стабильное потребление ресурсов и предотвращая лавинообразный рост нагрузки.

Комбинация этих политик формирует единый механизм отказоустойчивости: **Retry** обеспечивает восстановление соединения, **Circuit Breaker** — защиту от постоянных ошибок, **Timeout** — контроль зависших операций, а **Bulkhead** — защиту от перегрузки. В совокупности это гарантирует устойчивое и непрерывное взаимодействие с внешними биржевыми источниками данных.

# Механизм дедупликации
Механизм дедупликации реализован по принципу **скользящего окна (sliding window)** с использованием потокобезопасных структур данных.

Каждый входящий тик преобразуется в уникальный ключ с помощью `GetHashKey`, который комбинирует основные поля (`Symbol`, `Price`, `Volume`, `EventTime`). Далее ключ проверяется в `ConcurrentDictionary`: если он уже существует — тик считается дубликатом, если нет — добавляется и обрабатывается как уникальный.

Для ограничения объёма памяти используется кольцевой буфер (ring buffer) фиксированного размера. Индекс буфера увеличивается атомарно через `Interlocked.Increment`, а при переполнении старые ключи вытесняются и удаляются из словаря. Это обеспечивает хранение только актуального набора элементов.

Таким образом, реализуется потокобезопасная, производительная дедупликация с ограниченным использованием памяти и эффективным удалением устаревших данных.

Дополнительно проверяется, что система:

## Описание property-based тестов

Property-based тестирование применяется для проверки общих свойств (инвариантов) системы на большом количестве автоматически сгенерированных входных данных. В отличие от классических unit-тестов, где задаются конкретные входные значения, данный подход позволяет проверить корректность поведения алгоритма в широком диапазоне сценариев.

В рамках механизма дедупликации property-based тесты направлены на проверку следующих свойств:

- отсутствие исключений при любых допустимых входных данных;
- детерминированность поведения (одинаковые входные данные → одинаковый результат);
- корректное определение дубликатов при повторной обработке одинаковых тиков;
- устойчивость к некорректным значениям (например, `NaN`, `Infinity`);
- корректная работа в условиях многопоточности;
- изоляция состояния между различными экземплярами;
- стабильное поведение при переполнении скользящего окна.

Данный подход позволяет выявлять ошибки, которые сложно обнаружить с помощью заранее подготовленных тестовых сценариев, и значительно повышает надёжность системы за счёт проверки её поведения на случайных и граничных данных.

## Описание property-based тестов

Property-based тестирование используется для проверки общих свойств (инвариантов) системы на большом количестве автоматически генерируемых входных данных. В отличие от классических unit-тестов, где проверяются заранее заданные сценарии, данный подход позволяет исследовать поведение системы на широком диапазоне значений, включая граничные и случайные случаи.

Основная идея заключается в том, что проверяется не конкретный результат для конкретного входа, а свойство, которое должно выполняться всегда вне зависимости от входных данных.

В контексте Dataflow-пайплайна property-based тесты позволяют проверить, что:

- пайплайн всегда завершает выполнение при любом количестве входных данных;
- в систему сохраняются только уникальные данные;
- размер батчей не превышает заданные ограничения;
- метрики остаются согласованными и не нарушают инварианты;
- система корректно обрабатывает случайные и граничные значения;
- поведение системы остаётся стабильным при различных комбинациях входных параметров.

Использование property-based тестирования повышает надёжность системы, так как позволяет выявлять ошибки, которые сложно обнаружить с помощью заранее подготовленных тестовых сценариев, и подтверждает корректность работы системы в широком диапазоне условий.

## Описание нагрузочного теста

Нагрузочный тест предназначен для оценки производительности и устойчивости Dataflow-пайплайна при обработке большого объёма потоковых данных.

В ходе теста генерируется значительное количество тиков (до 1 000 000), которые последовательно поступают в систему через WebSocket-клиент. Измеряется общее время выполнения обработки с помощью `Stopwatch`, что позволяет определить фактическую пропускную способность системы.

Основные цели теста:

- проверка корректности обработки больших объёмов данных;
- оценка производительности системы;
- выявление узких мест в обработке;
- подтверждение способности системы масштабироваться под нагрузкой.

В рамках проверки анализируются следующие показатели:

- количество успешно обработанных и сохранённых тиков;
- время выполнения обработки;
- пропускная способность (тик/секунда).

Критерием успешного прохождения теста является достижение минимального порога производительности (например, более 10 000 тиков в секунду) и корректное завершение обработки без ошибок и потерь данных.

## Dataflow Pipeline (интеграционные тесты)

| Тест | Описание | Проверяемые аспекты | Ожидаемый результат |
|------|--------|---------------------|---------------------|
| Should_process_ticks_and_save_to_database | Проверяет полный цикл обработки тиков | Dataflow, репозиторий, метрики | Данные сохраняются, `In` и `Out` > 0 |
| Should_create_batches_with_correct_size | Проверяет корректное формирование батчей | Батчинг, ограничение размера | Размер батча в диапазоне 1–2000 |
| Should_remove_duplicates | Проверяет удаление дубликатов | Дедупликация, метрики | В БД нет дублей, `Deduplicated` > 0 |
| Should_handle_multiple_clients_with_load | Проверяет работу с несколькими источниками | Параллелизм, масштабирование | Обрабатывается большой поток, записи есть |
| Should_complete_gracefully | Проверяет корректное завершение пайплайна | Надёжность, завершение | Пайплайн завершается без ошибок |
| Should_deduplicate_across_clients | Проверяет дедупликацию между клиентами | Глобальная дедупликация | Нет дубликатов в БД |

Интеграционные тесты подтверждают корректную работу всей системы целиком: от получения данных из нескольких источников до их обработки, дедупликации и сохранения в базу, включая контроль метрик и устойчивость к нагрузке.

## Dataflow web socket client (unit-тесты)

| Тест | Описание | Проверяемые аспекты | Ожидаемый результат |
|------|--------|---------------------|---------------------|
| Should_Handle_Cancellation_Gracefully | Проверяет корректную остановку клиента при отмене | Обработка CancellationToken | Метод завершается без исключений |
| Should_LogError_When_Unexpected_Exception | Проверяет обработку ошибок | Логирование, fault tolerance | Ошибка логируется через `ILogger` |
| Should_Send_Converted_Items_To_Target | Проверяет передачу данных в pipeline | Интеграция connector, receiver, converter | Конвертированные `Tick` передаются в `BufferBlock` |

Данные тесты проверяют поведение WebSocket-клиента в составе пайплайна:
- устойчивость к отмене выполнения,
- корректную обработку ошибок и логирование,
- передачу преобразованных данных в следующий этап обработки.

## Default message receiver (unit-тесты)

| Тест | Описание | Проверяемые аспекты | Ожидаемый результат |
|------|--------|---------------------|---------------------|
| Should_Receive_And_Handle_Message | Проверяет получение текстового сообщения | Обработка WebSocket, вызов handler | Сообщение корректно передаётся в обработчик |
| Should_Ignore_Binary_Message | Проверяет игнорирование бинарных сообщений | Фильтрация типов сообщений | Handler не вызывается |
| Should_Log_Error_When_Handler_Fails | Проверяет обработку ошибок в handler | Логирование, устойчивость | Ошибка логируется, обработка продолжается |
| Should_Stop_When_Connection_Closed | Проверяет остановку при закрытии соединения | Обработка закрытия WebSocket | Чтение прекращается |
| Should_Stop_When_Receive_Fails | Проверяет поведение при ошибке чтения | Обработка исключений | Цикл завершается без вызова handler |
| Should_Throw_When_Message_Too_Large | Проверяет защиту от переполнения буфера | Ограничение размера сообщения | Генерируется `InvalidOperationException` |

Тесты `DefaultMessageReceiver` проверяют корректную работу компонента, отвечающего за приём сообщений из WebSocket:

- обработку текстовых сообщений и их передачу в обработчик;
- игнорирование неподдерживаемых (бинарных) сообщений;
- устойчивость к ошибкам внутри обработчика;
- корректное завершение работы при закрытии соединения;
- обработку ошибок чтения;
- защиту от переполнения буфера.
